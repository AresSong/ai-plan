    Evidence Snapshot: Why Local Evaluation Matters

    Clear upside exists GitHub reported Copilot users completed a simple programming task about 55% faster.
    Quality can improve A GitHub study of 202 valid submissions found Copilot users were 53.2% more likely to pass all unit tests.
    But context can reverse gains METR reported 16 experienced developers were 19% slower on 246 realistic tasks in familiar projects.
    Enterprise signals vary ZoomInfo reported 33% suggestion acceptance, 20% line acceptance, and 72% developer satisfaction.
    DORA shows tradeoffs Higher AI adoption correlated with better docs, code quality, and reviews, but lower throughput and stability.
    Security needs its own gate The exported research cites a Veracode report where about 45% of tested AI-generated code had security flaws.
    Best first eval suite Start with 20 to 30 real tasks: recurring work, hard tasks, and failure or security cases.
    Useful local tooling Use Promptfoo for fast matrix runs, Inspect for deep agent benchmarks, and Phoenix, Langfuse, or MLflow for traces.

    Minimum Viable Operating Loop

    1. Baseline Record the current model, instructions, tools, workflow, and normal task outcomes.
    2. Benchmark Build 20 to 30 representative tasks from real bugs, features, reviews, and migrations.
    3. Compare Run candidate stacks on the same tasks with tests, review checks, cost, and latency captured.
    4. Promote Move a stack to default only after quality, safety, cost, and speed gates are met.
    5. Monitor Track weekly success rate, rework, failures, review burden, incidents, and cost per useful task.
    6. Learn or Roll Back Turn failures into evaluation cases, improve the stack, or restore the prior manifest.

    Detailed Capabilities for the Q&A

    1. Measurable value Outcome-led engineering plus human feedback and accountability.
    2. Reproducible system Manage the whole system, not just the model, with reproducibility and lineage.
    3. Evidence-based change Experimentation, evaluation, registry, promotion, rollback, and telemetry.
    4. Safe adaptation Drift response, continuous improvement, governance, security, and trust.
    Stack manifest Model, settings, repo SHA, instructions, prompts, skills, tools, guardrails, environment, and eval suite.
    Run registry Task ID, stack ID, tool calls, generated diff, tests, review outcome, human edits, duration, and cost.
    Promotion gate Promote only when task success, quality, safety, review burden, latency, and cost clear the bar.
    Learning loop Rejected diffs, incidents, drift, and painful reviews become new eval cases or stack improvements.
